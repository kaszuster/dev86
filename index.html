Radio-API

In diesem Projekt werden Daten aus unterschiedlichsten Quellen (FTP, Rest, SOAP, Pull, Push, RMI Clients, Live Events aus dem Studio, etc.) und Formaten angeliefert. Eine Vielzahl an Importern, die isoliert betrieben werden, werten diese Daten aus und persistieren sie in optimierten Form in einer NoSQL Datenbank. Verschiedene Exporter arbeiten mit diesen Daten und spielen sie gezielt Drittsystemen zu. Die Importer und Exporter arbeiten asynchron und nahezu in Echtzeit. Unter Berücksichtigung hoher Lastspitzen bietet eine Rest API Zugriff auf diese Daten und ermöglicht den Wellen-Apps (Einslive, WDR2, etc.) sowie den Webseiten des WDR, Zugriff auf Playlisten und Sendepläne. Mehrere Bestandssysteme wurden durch diese API abgelöst, so wurde das Produkt zum zentralen System von Sendedaten (prospektiv) und Playlisten (retrospektiv), indem es sie konsolidiert, verarbeiten und verbreiten.

Eingesetzte Technologien & Frameworks: Microservices, Java 11, Maven, Git, Spring Boot, Apache Camel, Rest, Soap, Solr, EhCache, Akamai, Ascii Doc, PlantUml, Swagger, Scrum

WDR Webangebote und Mediendatenbank

Eine Vielzahl von Redakteuren aus den unterschiedlichsten Bereichen (Sport, Wirtschaft, Politik, etc.) pflegen Medieninhalte in der zentralen Mediendatenbank (ZMDB). Diese Inhalte bestehen aus Binärdaten zu Audios und Videos in verschiedenen Formaten, Qualitäten und Metadaten. Die hier gepflegten Daten werden an die Webausspielung (CMS und Akamai Edge Server) sowie Drittsysteme (ARD Mediathek) verteilt, um sie dem Endbenutzer über Web und App anzubieten. Das Webangebot wird auf Basis des Sophora CMS realisiert. Die umgesetzten Anforderungen in diesem Themenkomplex sind sehr vielseitig und unterschiedlich komplex: - Ausspielung neuer Pflegbarer Komponenten - Einführung neuer Dokumentstrukturen für Projekte wie dem Kandidatencheck - Konzeption und Entwicklung einer Google AMP Ausspielung in Form eines neuen Ausspielkanals aller Bestandsbeiträge - Planung, Entwicklung und Integration neuer Services wie dem Reaction Service, der Benutzern ermöglicht mittels Emojis auf Beiträge zu reagieren - Beratung des Architektur-Teams - Performance Optimierung - Schulung des Kollegiums in Form von Barcamps zu Themen wie Jenkins Pipeline, Git/Git Flow, Clean-Code und Problem der vorliegenden Architektur

Eingesetzte Technologien & Frameworks: Java 11, Maven, Spring, Sophora CMS, JCR, Solr, MySQL, JSP, Primefaces, XMLT, Groovy, Jenkins, CI/CD, CXF, Apache Tomcat, Docker

Referencer (über Adesso AG)

In diesem Projekt wurde ein Tool erstellt, das die Verwaltung und Suche von Projektreferenzen ermöglicht. Bis zur Entwicklung dieses Tools wurden Projektreferenzen in Word Dateien dokumentiert. Bei Angebotserstellung oder der Teilnahme an Ausschreibungen mussten Projektreferenzen mühsam in diesen Word Dateien gesucht und zusammengestellt werden. Der Referencer hat diesen Schritt optimiert, indem er die Daten über LDAP ausgelesen und in einer NoSQL Datenbank abgelegt hat. Die Facettensuche (Technologien, Branchen, Volumen, beteilige Rollen, Schlagworte, etc.) und Exportfunktion (PDF und Word Format), auf Basis verschiedener Vorlagen, konnte den Angebotsprozess in Qualität und Aufwand verbessern. Die Datenpflege erfolgt weiterhin in Word Dateien, ohne dass der bisherige Dokumentationsprozess anpasst werden musste.

Eingesetzte Technologien & Frameworks: Self-Contained System, Java 8, Spring Boot, Maven, Git, Angular, Elasticsearch, Apache POI, LDAP, Kanban
